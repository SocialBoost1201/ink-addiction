# INK ADDICTION LP – 運用・本番設計 v1.1

受領日: 2026-02-27  
ステータス: 確定（v1.0 → v1.1 差分を本書に統合済み）

---

## 1. 目的

本書は、INK ADDICTION LP の本番運用・保守・監視・バックアップ・障害対応・継続改善の設計を定義する。

本LPは「問い合わせの生命線」かつ「予約導線の中核」であるため、停止や誤公開は事業損失に直結する。  
**安定運用を最優先とする。**

---

## 2. 権限設計（v1.1更新）

### 2.1 ロール定義

| ロール            | 担当者      | 権限                                                       |
| ----------------- | ----------- | ---------------------------------------------------------- |
| **Owner**         | 兄 / Takuma | 全権限 / 公開操作 / スロット承認 / 設定変更 / 本番反映承認 |
| **Admin（将来）** | スタッフ    | 問い合わせ管理・作品編集のみ / 公開・設定変更不可          |

### 2.2 Owner のみ実行可能な操作

- 作品公開 / 非公開
- スクショ抽出承認
- スロット公開
- 設定変更
- 本番環境への反映

> **allowlistに兄 + Takuma の両メールアドレスを登録する。Owner削除はOwner同士でのみ可能。**

## 3. 環境構成

### 3.1 環境分離

| 環境        | 用途                 |
| ----------- | -------------------- |
| Development | 開発・デバッグ       |
| Staging     | 本番反映前の動作確認 |
| Production  | 本番                 |

各環境でDB・Storageを分離する。**本番データを開発環境へ直接接続しない。**

### 3.2 デプロイ方針

- `main` ブランチのみ本番デプロイ
- PRマージ前に必ずStaging確認
- 破壊的変更禁止
- DBマイグレーションは create-only

### 3.3 本番変更承認フロー

**デプロイ承認**

- `main` へのマージはOwnerのみ可能
- 本番デプロイ前にOwner確認
- 変更ログを監査テーブルへ保存

**設定変更**

- `settings` テーブル変更時は `admin_audit_logs` に必ず記録
- `updated_by` にOwnerのUUIDを保存

## 4. バックアップ設計

### 4.1 DBバックアップ

- 毎日自動バックアップ
- 30日保持
- 重点監視テーブル: `works` / `inquiries` / `availability_slots`

### 4.2 Storageバックアップ

- 作品画像: 二重保存不要（元データ保持）
- 問い合わせ添付: 定期エクスポート可能にする

### 4.3 CSVエクスポート機能

管理画面から以下をCSVエクスポート可能にする。

- `inquiries`
- `availability_slots`
- `works`

## 5. ログ管理

### 5.1 アプリログ

- エラー
- API失敗
- OCR失敗
- 画像分類失敗

### 5.2 監査ログ

- **削除不可**
- 最低180日保持
- v1.1追加: 重要操作のpayloadに `before_value` / `after_value` を含める
- v1.1追加: `approval_actor_role` / `approval_timestamp` を記録

### 5.3 ログ監視アラート

| 条件               | アクション   |
| ------------------ | ------------ |
| エラー急増         | アラート通知 |
| OCR失敗連続発生    | アラート通知 |
| `inquiry` 保存失敗 | 即時通知     |

---

## 6. 障害対応設計

### 6.1 問い合わせ送信障害

| 障害           | 対応                  |
| -------------- | --------------------- |
| DB保存失敗     | retry 3回             |
| メール送信失敗 | `email_outbox` へ保存 |
| 未送信残存     | 管理画面に警告表示    |

### 6.2 OCR失敗

- `extraction_status = failed` をセット
- 管理画面で手動スロット登録へ誘導

### 6.3 画像分類失敗

- `autoGenre = null` で保存
- 管理画面で手動設定を必須表示

### 6.4 DB接続障害

| 対象          | 対応                          |
| ------------- | ----------------------------- |
| Public ページ | キャッシュ表示                |
| 管理画面      | エラーメッセージ表示          |
| 重大障害時    | Inquiry停止・メンテナンス表示 |

## 7. 公開事故防止設計

### 7.1 技術的制御

| ルール                          | 実装箇所                           |
| ------------------------------- | ---------------------------------- |
| `finalGenre` 未設定では公開不可 | UIトグル無効化 + APIバリデーション |
| OCR未承認スロット非公開         | 承認フロー必須                     |
| `is_deleted = true` 作品非表示  | RLS                                |
| `booked` スロット非表示         | RLS / APIフィルタ                  |

### 7.2 公開直前確認

公開トグル操作時に確認モーダルを必ず表示する。

```
「この作品を公開しますか？」
```

## 8. セキュリティ運用

### 8.1 認証管理

- AdminメールのallowlistはOwnerのみ変更可能
- 不要アカウントは即削除
- パスワード再設定フローを定期確認

### 8.2 定期確認

- RLSポリシー検証
- Storage公開設定確認
- アクセスログ確認

### 8.3 添付ファイル運用

- MIME検証
- サイズ制限
- 実行可能ファイル（`.exe` 等）拒否

## 9. パフォーマンス監視

### 9.1 定期計測（月1回）

| 指標 | 基準      |
| ---- | --------- |
| LCP  | 2.5秒以内 |
| CLS  | 0.1以下   |
| TTFB | 600ms以内 |

### 9.2 画像監視

- 1枚10MB超過不可
- サムネ生成が正常に完了しているか確認

## 10. 運用フロー

### 10.1 作品追加フロー

```
1) 画像撮影
2) 管理画面アップロード
3) 自動分類確認（autoGenre + confidence）
4) finalGenre確定
5) 公開
```

### 10.2 空き状況更新フロー

```
1) スクショ取得
2) 管理画面アップロード
3) 抽出結果確認（confidence確認）
4) 手動編集（必要に応じて）
5) 承認・公開
```

### 10.3 問い合わせ対応フロー

```
1) 通知受信
2) 内容確認
3) status = reviewing
4) 返信
5) scheduled または closed
```
