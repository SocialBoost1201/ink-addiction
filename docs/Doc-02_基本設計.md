受領日: 2026-02-27  
ステータス: 確定

---

## 1. 基本設計の位置づけ

本書は要件定義書で定義した機能を、画面構成・導線・主要コンポーネント・主要データ・処理フロー・権限制御・外部連携の観点で「実装可能な粒度」に落とし込む。  
本LPは情報サイトではなく、ギャラリーと予約問い合わせを中心とした軽量運用基盤である。

---

## 2. システム全体像

### 2.1 構成要素

| 構成                             | 役割                                                                                                   |
| -------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **A. Public Web（LP本体）**      | 作品閲覧、スタイル探索、空き状況閲覧、問い合わせ送信、SNS遷移                                          |
| **B. Admin Console（管理画面）** | 作品アップロード・分類調整、ジャンル管理、スクショ解析承認、空き枠公開管理、問い合わせ管理、設定       |
| **C. Worker（非同期処理）**      | 作品画像の自動ジャンル分類 / スケジュールスクショのOCR解析と空き候補抽出 / 画像最適化メタデータ生成    |
| **D. Data Layer**                | DB（作品・ジャンル・空き枠・問い合わせ・監査ログ） / Storage（作品画像・問い合わせ添付・スクショ原本） |

### 2.2 データの責務分離

- 表示用メタデータはDB、実ファイルはStorage
- 自動推定結果は保持するが、**公開値は必ず管理者が最終確定する**

## 6. 主要機能の処理フロー

### 6.1 作品アップロードと自動分類

```
1) 管理者が作品を作成し画像をアップロード
2) Storageに保存、DBにメタ登録
3) Workerが画像を取得しジャンル推定とタグ推定
4) 推定結果をDBへ保存（autoGenre, autoTags, confidence）
5) 管理者が確認し確定値を保存（finalGenre, finalTags）
6) 公開ページに反映
```

> **必須ルール:** autoは参照用。公開はfinalのみ。final未設定の場合は「要確認」扱いで非公開も可能にする。

### 6.2 スクショ取り込みと空き候補抽出

```
1) 管理者がスクショをアップロード
2) WorkerがOCRし予定ブロックを抽出
3) 空き候補スロットを生成し、confidenceを付与
4) 管理画面で候補を編集（削除、時間修正、追加）
5) 管理者が承認し公開
6) 公開スロットが /availability に表示
```

> **必須ルール:** 自動公開禁止。承認後のみ公開。公開スロットのステータスは `published / hidden / booked` で管理。

### 6.3 問い合わせ送信

```
1) ユーザーがマルチステップで入力
2) 画像をアップロード（任意）
3) 送信時にDBへInquiry作成
4) 管理者へ通知（メール）
5) ユーザーへ自動返信（受付内容、次の流れ、目安返信時間）
6) 管理画面でステータス更新と対応履歴を残す
```

### 6.4 スパム対策

- レート制限
- honeypot
- 添付のMIMEと拡張子検査
- サイズ上限
- 同一IPの連投抑止
- 設定でCAPTCHAをONにできる

## 7. データ設計の基本方針

### 7.1 データモデル（概要）

```
Works / WorkImages / Genres
AvailabilitySlots / ScheduleScreenshots
Inquiries / InquiryUploads
AdminAuditLogs
```

### 7.2 監査ログ対象

作品公開 / ジャンル変更 / スロット公開 / 問い合わせステータス変更

### 7.3 削除方針

- 作品: 原則ソフト削除

## 10. SEO基本設計

| 対象                  | 方針                                                       |
| --------------------- | ---------------------------------------------------------- |
| `/styles/{genreSlug}` | SEOの柱。ジャンル別ページをインデックスの主軸とする        |
| `/gallery/{workId}`   | インデックス対象。説明文とタグを必須化し薄いページにしない |
| `/faq`                | 長尾獲得                                                   |
| 構造化データ          | `LocalBusiness` / `FAQPage` / `ImageObject` を適用         |

---

## 11. パフォーマンス基本設計

- 画像は必ず最適化し、サムネと高解像の段階読み込み
- ギャラリーは無限スクロールでも初期描画を軽くする
- `/gallery` はフィルタ条件をURLパラメータで保持し共有可能にする
- 管理画面はデータテーブルをページネーションにし一括取得しない

---

## 12. エラーハンドリング基本設計

- アップロード失敗は再試行可能にし、途中入力を保持
- スクショ解析失敗は手動入力で救済できる
- 問い合わせ送信は二重送信防止（idempotency）

---

## 13. v1 スコープ確定

### 含む

### 含まない

- 自動予約確定
- 決済
